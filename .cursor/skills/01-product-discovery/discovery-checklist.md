# Discovery Phase — Human Review Gate Checklist

Use this checklist before approving the transition from Product Discovery → Product Design.

---

## PDLC Framework Selection
- [ ] PDLC framework selected and documented in PDLC Selection Record
- [ ] Selection rationale references product type, team capacity, risk profile, and timeline
- [ ] Phase scope adjustments (compressed/skipped phases) are documented with risk notes
- [ ] Stage-gate entry and exit criteria are defined
- [ ] Approver for the discovery gate is identified

---

## Stakeholder Alignment
- [ ] Business goals are clearly stated and agreed upon by stakeholders
- [ ] Success metrics are specific and measurable
- [ ] Constraints (technical, budget, timeline) are documented
- [ ] Non-goals are explicitly listed to prevent scope creep
- [ ] All open questions from stakeholder brief have been addressed or assigned

## Problem Definition
- [ ] Problem statement is a single, clear sentence (user + need + insight)
- [ ] The problem is validated — it's real, not assumed
- [ ] HMW questions are open-ended and cover the opportunity space
- [ ] Root cause of the problem is understood (not just symptoms)

## User Research
- [ ] Research was conducted with real users OR assumptions are clearly marked
- [ ] At least 2–3 distinct user perspectives were considered
- [ ] Surprising or contradictory findings are documented
- [ ] Research insights are grounded in evidence (quotes, observations)
- [ ] Assumptions requiring future validation are listed

## Competitive Analysis
- [ ] At least 3 direct competitors analyzed
- [ ] At least 2 indirect competitors analyzed
- [ ] Differentiation opportunities are clearly identified
- [ ] Pricing and business model landscape is understood
- [ ] Deep competitive intelligence applied (changelog, review sentiment, pricing archaeology, job listings) where warranted

## Strategic Positioning
- [ ] Blue Ocean ERRC Grid completed with at least 2 items per quadrant (or quadrant explicitly marked as none-applicable)
- [ ] Strategy Canvas plotted for product and at least 3 competitors across 6–10 factors
- [ ] Three tiers of non-customers identified and described
- [ ] Value Proposition Canvas completed for primary persona — customer profile and value map both filled
- [ ] Fit Score calculated and is ≥ 70% (or gap items are documented and accepted by stakeholder)
- [ ] Kano Model applied to top 8–10 candidate features — Basic / Performance / Excitement classified
- [ ] Porter's Five Forces assessment completed with intensity ratings and strategic implications
- [ ] USP Statement drafted using the differentiation ladder (at Value or Belief level)
- [ ] USP passes all four validation tests (differentiation clarity, believability, relevance, competitor gap) — or failures are documented with remediation plan

## Personas
- [ ] 2–3 personas created
- [ ] Each persona has clear goals, frustrations, and behaviors
- [ ] Personas are distinguishable from each other (not generic archetypes)
- [ ] Primary persona is identified
- [ ] Any AI-generated assumptions are flagged for validation

## Data Dictionary (Conditional)
- [ ] Data Dictionary trigger check completed — document whether it applies (Yes / No / Not applicable)
- [ ] If triggered: all core entities identified and named
- [ ] If triggered: key attributes defined with type, business rules, and constraints
- [ ] If triggered: PII / sensitive fields explicitly flagged
- [ ] If triggered: entity lifecycle states documented
- [ ] If triggered: data ownership assigned per entity
- [ ] If triggered: open questions and assumptions logged with owners and due dates
- [ ] If not triggered: rationale documented ("No complex data model / No regulated domain / No multi-system integration")

## Journey Mapping
- [ ] Current state journey mapped for primary persona
- [ ] Key pain points identified and ranked by severity
- [ ] Future state journey sketched
- [ ] Critical moments of truth identified

## PRD
- [ ] Problem statement matches the problem definition artifact
- [ ] All user stories follow "As a / I want / So that" format and are grounded in JTBD job statements
- [ ] P0 requirements are minimal and truly must-have
- [ ] All functional requirements are specific and testable
- [ ] Non-functional requirements include accessibility (WCAG 2.1 AA) and performance
- [ ] Kano Basic features are all represented in P0 requirements
- [ ] At least one Excitement (delighter) feature is included as a differentiator
- [ ] USP is reflected in the Executive Summary and goals
- [ ] If Data Dictionary was produced: data requirements in PRD reference entity definitions
- [ ] Success metrics are defined and measurable
- [ ] Risks and mitigations are documented
